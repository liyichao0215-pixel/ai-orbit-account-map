# AI Orbit 五角色模拟可用性测试报告

> 测试日期：2026-07-23（Asia/Shanghai）  
> 测试对象：本地 AI Orbit 账号关系可视化原型  
> 测试地址：`http://localhost:3000/`  
> 测试性质：5 个 Agent 按角色假设完成的真实页面任务走查与启发式评估。它不是 5 位真人访谈，不能替代真实用户研究，也不能被写成真实业务验证。

> 版本说明：本报告记录的是 V1.0 基线问题。V1.1 已修复匿名账号二次打分、235/249/720 数量口径、数据快照入口和“立即建联”措辞，并增加本机运营清单。报告中的相关表述保留为迭代证据，不代表当前版本仍有这些问题。

## 0. V1.1 整改回读

| V1.0 发现 | V1.1 处理 | 当前口径 |
|---|---|---|
| 匿名占位简介产生 110 个虚假 S/A | 新增公开展示评分层，匿名字段不再二次打分 | S 44、A 81，共 125 个公开候选 |
| 235、249、720 三套数量混用 | 候选数与图谱上下文分开统计 | 125 个候选 + 14 个关联官号 |
| “立即建联”超过证据强度 | 改为“优先人工核验 / 进入复核名单” | 评分只支持初筛 |
| 数据快照按钮不可用 | 增加完整时间、来源、数据集与规则版本 | snapshot + UTC+8 |
| 缺少运营闭环 | 增加本机清单、负责人、状态、备注、下一步和 CSV | 仍不包含多人协作或自动触达 |

## 1. 执行结论

当前版本适合作为：

- AI 产品运营 / AIGC 产品运营岗位的个人作品集原型；
- “账号关系发现 + 可解释候选初筛”的内部探索工具；
- 展示产品拆解、数据建模、隐私分层、可视化和迭代判断的面试案例。

当前版本不适合作为：

- 可直接执行的达人建联名单；
- 每日矩阵账号运营主看板；
- 达人 CRM；
- 已经得到真实用户或业务结果验证的成熟产品。

五个模拟角色的启发式综合评分平均为 **6.2/10**。评分不是统计学研究结果，只用于比较不同角色的关注点。

产品最明确的价值是：

> 从多个核心 AI 官号的关注关系中，发现被多品牌共同关注的创作者，并解释为什么值得优先人工研究。

最需要优先处理的不是继续增加 3D 特效，而是：

1. 修复 S/A 分级、脱敏状态和页面承诺不一致；
2. 统一 235、249、720 三套数量口径；
3. 补充数据时间、来源、规则版本和评分证据；
4. 增加“加入名单—人工复核—分配—联系—结果回写”的最小运营闭环；
5. 再用真实运营人员验证任务成功率和排序效果。

## 2. 测试方法

### 2.1 五个模拟角色

| 角色 | 核心工作目标 | 重点观察 |
|---|---|---|
| 海外矩阵号运营负责人 | 建立候选发现、账号分组和团队工作流 | 产品价值、数据可信度、能否进入团队流程 |
| 达人合作 / 创作者生态运营 | 发现、筛选、邀约和复盘达人合作 | 地区、语言、内容、互动、联系方式、合作适配 |
| 新入职运营实习生 | 无培训完成筛选和解释任务 | 新手理解、关系方向、术语、状态反馈 |
| 矩阵号一线运营专员 | 每日找人、安排动作、交接和复盘 | 操作效率、保存、负责人、状态、导出 |
| AI 产品运营 / 数据产品经理 | 审核指标、评分、隐私和验收 | 数据契约、规则版本、模型效果、稳定性 |

### 2.2 统一任务

每个 Agent 都需要实际操作页面完成：

1. 从首屏判断产品用途、数据时间和隐私边界；
2. 找到并复述 S/A 资格门槛、计分公式和分级阈值；
3. 使用 S/A 筛选和候选雷达选择一个账号；
4. 使用搜索、3D 图或详情判断共享官号关系；
5. 判断详情证据能否支持该角色的下一步动作；
6. 判断产品能否进入该角色的日常工作流。

### 2.3 测试限制

- 这是角色化模拟和启发式预检，不代表真人用户态度。
- Agent 使用自动化浏览器操作，工具调用时间不等于页面性能，因此本报告不提供可靠的加载耗时。
- 没有验证真实邀约回复率、合作率、内容表现或业务收益。
- 没有进行多人协作、长期留存或移动端真人测试。

## 3. 已验证的页面事实

### 3.1 全量数据

- 1,809 个账号；
- 16 个关系社区；
- 16 个核心官号；
- 2,410 条关注关系；
- 页面显示的源数据时间为 `7/15 16:20`。

### 3.2 S/A 规则

资格门槛：

- 粉丝量在 2,000–5,000,000 之间；
- 内容匹配至少 8 分；
- 未通过门槛时进入 WATCH，不参与 S/A 分级。

总分：

```text
总分 = 粉丝影响力（最高 45）
     + 核心官号共同关注（最高 35）
     + 简介内容匹配（最高 20）
```

分级：

- S：总分 ≥ 75；
- A：65–74；
- B：55–64；
- 其余为 C 或 WATCH。

简介内容匹配信号：

- AI 相关：+9；
- 创作人：+8；
- 行业建设者：+5；
- 最高 20 分。

### 3.3 实测候选：INK

INK 在综合优先排序中位于第一：

- 13.6 万粉丝；
- 被 5 个核心官号共同关注；
- 影响力 33/45；
- 品牌共识 35/35；
- 内容匹配 20/20；
- 总分 88，S 级；
- 可跳转真实 X 主页。

共同关注官号包括：

- Seedream / Seedance；
- Kling；
- Hailuo；
- Lovart；
- OiiOii。

五个角色都认可：这些信息足以支持“优先人工研究”，但不足以直接支持“立即建联”。

## 9. 跨角色问题一：匿名 S/A 与公开承诺冲突

### 9.1 页面证据

页面承诺：

> 官方账号及最终 S/A 级账号保留公开身份；B 级及以下继续脱敏。

但候选榜实际出现：

- “创作者账号 0202”；
- `@account_0202 · 匿名标识`；
- 11 万粉丝；
- 5 个品牌共同关注；
- S 级，84 分；
- 页面写“立即建联”；
- 没有真实主页链接。

### 9.2 数量影响

| 层级 | 当前总数 | 匿名数 | 公开身份数 |
|---|---:|---:|---:|
| S | 55 | 11 | 44 |
| A | 180 | 99 | 81 |
| B | 485 | 485 | 0 |
| S/A 合计 | 235 | 110 | 125 |

当前 S/A 中有 **110 个匿名账号，占 46.8%**。

这不是身份泄露：匿名账号仍未暴露真实身份。但它是严重的数据契约和产品承诺问题，因为匿名账号不能执行建联动作。

### 9.3 实现侧根因交叉核验

当前管道的行为是：

1. 脱敏脚本先对原始数据计算 S/A；
2. 原始 S44、A81 保留公开身份；
3. B/C/WATCH 被匿名化并扰动精确指标；
4. 匿名账号的统一简介包含 `AI creator` 等评分关键词；
5. 前端再次对脱敏后的字段重算评分；
6. 部分匿名账号因此被重新提升为 S/A。

也就是说，身份是否公开使用了脱敏前的 tier，而雷达使用了脱敏后的重算 tier，两者不是同一个最终数据契约。

### 9.4 严重级

**P1，且应作为“建联看板”发布前阻断条件。**

### 9.5 验收标准

- 定义唯一 `final_tier`；
- 定义唯一 `identity_policy`；
- 写入 `scoring_rule_version`；
- 写入 `privacy_transform_version`；
- 展示扰动字段不得参与重新排名；
- 若产品承诺 S/A 公开，则 `anonymous S/A = 0`；
- 无法公开身份的高分账号进入“待身份核验”，不得显示“立即建联”；
- 相同输入和相同规则版本重复生成时，tier 数量和排序完全一致；
- 页面统计必须由最终展示数据动态生成。

## 10. 跨角色问题二：235、249、720 三套数字

页面同时出现：

- S 55 + A 180 = 235 个 S/A 候选；
- S/A 优先图中 249 个节点；
- 雷达中 720 个 B 级以上候选。

实际集合是：

```text
235 个 S/A 候选
+ 14 个提供关系上下文的核心官号
= 249 个图中节点

235 个 S/A
+ 485 个 B
= 720 个 B 级以上候选
```

计算本身成立，但页面没有解释集合关系。

### 严重级

**P1：影响数据可信度和任务理解。**

### 建议文案

```text
235 个 S/A 候选 + 14 个关联官号 = 249 个图中节点
```

雷达另行明确：

```text
全部 B 级以上候选 720：S 55 / A 180 / B 485
```

## 11. 跨角色问题三：评分可解释，但没有效果验证

当前评分适合作为产品假设，不适合作为客观价值判断。

风险包括：

- 简介写到 AI，不代表近期持续发布 AI 内容；
- “creator”关键词不等于合作质量；
- 共同关注代表关系，不代表品牌背书；
- 粉丝量不等于互动质量；
- “立即建联”把规则输出写成了行动结论。

建议将动作标签改为：

- S：优先人工核验；
- A：进入复核名单；
- B：观察；
- 匿名高分：待身份核验。

### 排序效果实验

准备至少 100 个候选，由两名真实运营者独立标注：

- 值得复核；
- 可建联；
- 不适合。

对比三组排序：

- A：仅粉丝量；
- B：粉丝量 + 共同关注；
- C：当前三因素评分。

第一阶段建议目标：

- Precision@20 ≥ 70%；
- Top20 人工接受率比粉丝量基线提升 ≥ 20%；
- 双人标注一致率 Cohen's Kappa ≥ 0.6。

## 12. 跨角色问题四：缺少数据血缘和新鲜度

页面只显示 `7/15 16:20`，无法判断：

- 年份；
- 时区；
- 是采集、生成还是刷新时间；
- 数据采集区间；
- 来源覆盖；
- 缺失率；
- 使用的评分和脱敏规则版本。

建议数据快照面板至少展示：

- `dataset_id`；
- `collected_at_start`；
- `collected_at_end`；
- `generated_at`；
- `refreshed_at`；
- `timezone`；
- `source_scope`；
- `coverage`；
- `scoring_rule_version`；
- `privacy_transform_version`；
- 数据 hash 或构建编号。

## 13. 跨角色问题五：发现之后没有运营闭环

五个角色一致认为，当前产品完成：

```text
发现候选 → 理解关系
```

但没有完成：

```text
加入名单 → 人工复核 → 分配负责人 → 建联
→ 记录回复 → 合作结果 → 回写评分
```

V2 的最小范围应是：

1. 加入候选名单；
2. 负责人；
3. 状态；
4. 备注和下一步；
5. 保存筛选视图；
6. CSV / JSON 导出；
7. 结果回写。

## 14. 3D 图的真实产品价值

3D 图的优点：

- 第一眼有记忆点；
- 能表现社区、密度和多品牌关系；
- 适合探索和面试演示。

3D 图的限制：

- 新手不能仅靠线条理解关系方向；
- 节点密集时不适合精确比较；
- 搜索、列表和详情已经承担大部分任务；
- 目前没有证据证明 3D 比二维图或表格更高效。

建议保留 3D，但把它定位为“关系探索视图”，同时提供：

- 常驻图例；
- 明确方向；
- 二维列表；
- 可复制的关系解释；
- “返回全图”；
- 3D 与二维 / 表格的任务效率对照测试。

## 17. 面试使用建议

### 17.1 合理定位

适合定义为：

> 一个围绕 AI 官号关注关系构建的账号生态发现原型。我将公开账号关系建模成图谱，用球形社区展示关系聚合，并设计可解释的候选评分和分层隐私规则。第一轮用五个角色化 Agent 完成模拟任务走查，发现了评分、脱敏和统计口径不一致，以及发现到执行之间的产品断点。

### 17.2 不应使用的表述

- “访谈了 5 位运营人员”；
- “已经验证运营价值”；
- “S 级代表一定值得合作”；
- “产品已经可以管理矩阵账号”；
- “产生了建联率或合作转化提升”。

### 17.3 推荐面试讲述

> 我先复刻了 3D 账号关系图，但没有停在视觉层。  
> 我把目标重新定义为“帮助运营从官号关注网络中找到值得人工复核的候选”，并设计了粉丝影响力、共同关注和内容匹配三因素评分。  
> 随后我安排五个角色化 Agent 实际完成统一任务，把它当作启发式预检，而不冒充真人研究。  
> 测试发现 235 个 S/A 候选中有 110 个仍被匿名，说明评分重算与脱敏时点不一致；同时 249 的筛选数字混入了 14 个关系官号。  
> 我据此把迭代优先级从继续优化 3D，调整为统一数据契约、补快照血缘、修复数量口径和建立运营状态闭环。  
> 下一步我会用 3–5 位真实运营者验证任务成功率、Precision@20 和人工接受率。

## 18. 最终判断

这个项目用于面试是合理的，但应该把它定位成：

> **经过角色化启发式测试、已经识别关键数据契约问题的账号关系发现 MVP。**

它目前最能证明的能力是：

- 把模糊运营想法变成产品问题；
- 建立节点、关系、社区和评分数据模型；
- 做可解释的交互原型；
- 识别隐私和数据口径风险；
- 从测试证据调整迭代优先级；
- 对尚未验证的业务效果保持诚实。

如果完成 V1.1，并补 3–5 位真实运营者任务测试，它会从“好看的学习原型”升级为更有说服力的 AI 产品运营作品集案例。
